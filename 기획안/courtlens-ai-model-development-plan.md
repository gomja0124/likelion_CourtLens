# CourtLens AI 모델 개발 계획안

## 1. 문서 목적

이 문서는 CourtLens에서 프론트엔드/백엔드 구현과 별도로 개발해야 하는 AI 모델 영역을 정리한다.

CourtLens의 AI 모델은 크게 두 축이다.

1. 경기 영상을 보고 선수별 하이라이트 후보를 추출하는 영상 분석 모델
2. 팬 행동/댓글 반응을 보고 구단 대시보드에서 이벤트와 마케팅 액션을 제안하는 LLM 기반 전략 모델

아이디어톤 MVP에서는 두 모델을 완전히 학습시키는 것이 목표가 아니다. 목표는 어떤 데이터를 모으고, 어떤 모델 구조로 개발하며, 어떤 산출물을 프론트/백엔드와 연결할지 명확히 설명하는 것이다.

## 2. 전체 AI 구조

CourtLens의 AI 파이프라인은 다음 흐름으로 동작한다.

```text
경기 영상 / play-by-play / 선수 데이터
        ↓
영상 분석 모델
        ↓
선수별 하이라이트 후보 + 장면 태그 + 타임코드
        ↓
팬 앱에서 개인화 하이라이트 생성
        ↓
팬 행동 로그 / 댓글 / 저장 / 공유 / 클릭 데이터 누적
        ↓
LLM 기반 AI Strategy Engine
        ↓
구단 대시보드 Revenue Playbook / Event Builder / Sponsor Package / Performance 리포트
```

영상 모델은 "어떤 장면을 어떤 선수의 하이라이트로 잘라낼 것인가"를 담당한다.

LLM 모델은 "그 장면과 팬 반응을 구단이 실행할 수 있는 이벤트/마케팅/스폰서 제안으로 어떻게 바꿀 것인가"를 담당한다.

## 3. 모델 1: 선수별 하이라이트 추출 영상 모델

### 3.1 목표

경기 전체 영상 또는 구단이 제공한 경기 클립을 입력받아, 선수별로 의미 있는 하이라이트 후보를 자동으로 추출한다.

최종 산출물은 단순 영상 컷이 아니라 다음 정보를 포함해야 한다.

- 하이라이트 시작/종료 타임코드
- 관련 선수
- 관련 팀
- 장면 유형
- 장면 중요도 점수
- 초보자용 설명에 필요한 태그
- 작전판AI 설명에 필요한 전술 맥락

예시 산출물:

```json
{
  "clip_id": "LG_20260627_Q2_0532_YOO_3PT",
  "game_id": "KBL_20260627_LG_KT",
  "player_id": "yoo_gisang",
  "player_name": "유기상",
  "team": "창원 LG",
  "start_time": "00:38:12",
  "end_time": "00:38:27",
  "event_type": "transition_three_point",
  "highlight_score": 0.91,
  "tags": ["3점", "트랜지션", "팬 반응 후보", "외곽 슈터"],
  "explain_context": {
    "easy": "빠른 공격 상황에서 수비가 정리되기 전에 3점을 성공한 장면",
    "tactical": "트랜지션 상황에서 코너 수비 로테이션이 늦어진 틈을 활용"
  }
}
```

### 3.2 입력 데이터

필수 입력:

- 경기 영상 파일 또는 스트리밍 VOD URL
- 경기 일정/상대팀/홈·원정 정보
- 팀 로스터와 선수 등번호
- play-by-play 데이터
- 박스스코어

고도화 입력:

- 중계 자막 또는 ASR 결과
- 스코어보드 OCR 결과
- 좌표 기반 트래킹 데이터
- 구단 제공 주요 장면 태그
- 팬 앱에서 생성된 기존 하이라이트 반응 데이터

### 3.3 처리 파이프라인

초기 개발 방향은 "영상만 보고 모든 선수를 자동 인식하는 모델"이 아니라, KBL 공식 홈페이지의 문자중계/play-by-play를 기준 데이터로 사용해 경기 영상과 매칭하는 방식이 현실적이다.

KBL 경기 기록 페이지의 문자중계에는 쿼터와 경기 시계 기준으로 어떤 선수가 득점, 슛, 파울, 리바운드, 어시스트를 했는지에 대한 이벤트 로그가 있다. 이 데이터를 전체 경기 영상의 스코어보드 시간과 맞추면, 영상에서 득점으로 이어지는 구간을 자동으로 잘라 하이라이트 후보로 만들 수 있다.

1. 영상 전처리
   - 영상 프레임 샘플링
   - 스코어보드 영역 탐지
   - 경기 시간 OCR
   - 쿼터/시간 기준으로 영상과 play-by-play 동기화

2. 장면 분할
   - 득점, 리바운드, 어시스트, 스틸, 블록, 턴오버 같은 이벤트 주변 구간 추출
   - play-by-play 기준으로 이벤트 전후 8~12초 클립 후보 생성
   - 하이라이트용으로 불필요한 중계 전환, 리플레이, 광고 컷 제거

3. 선수 인식
   - 선수/심판/공 위치 탐지
   - 선수 트래킹
   - 등번호 OCR 또는 유니폼 특징 인식
   - 로스터 데이터와 매칭

4. 액션 인식
   - 슛, 3점, 덩크, 블록, 리바운드, 어시스트, 속공 등 장면 유형 분류
   - 득점 성공 여부와 play-by-play 이벤트를 함께 검증
   - 팀 공격/수비 상황 태그 생성

5. 하이라이트 점수화
   - 경기 이벤트 중요도
   - 선수 관심도
   - 희소성
   - 득점 기대 효과
   - 팬 반응 예측 점수
   - 구단 수익화 연결 가능성

6. 클립 생성
   - 시작/종료 타임코드 보정
   - 선수별 클립 묶음 생성
   - 썸네일 후보 프레임 생성
   - 팬 앱에 제공할 metadata JSON 생성

### 3.4 KBL 문자중계 기반 PoC 설계

이 프로젝트에서 가장 현실적인 1차 PoC는 `KBL 문자중계 이벤트 로그 + 전체 경기 영상 + FFmpeg 컷팅` 구조다.

완전한 선수 인식 모델을 먼저 만들면 난이도가 높다. 선수 얼굴/등번호/유니폼 인식은 카메라 각도, 중계 화질, 리플레이, 가림 현상 때문에 안정화가 어렵다. 반면 문자중계에는 이미 선수명과 이벤트 타입이 들어 있으므로, 초기에는 이 데이터를 "정답에 가까운 이벤트 로그"로 사용한다.

#### 3.4.1 입력 예시

문자중계 이벤트:

```json
{
  "game_id": "S47G01N256",
  "game_date": "20260401",
  "quarter": 2,
  "game_clock": "06:34",
  "team": "창원 LG",
  "player_name": "유기상",
  "event_type": "3점슛 성공",
  "score_after": "45-39"
}
```

전체 경기 영상:

```json
{
  "video_id": "S47G01N256_full_game",
  "video_path": "videos/S47G01N256.mp4",
  "quarter_start_offsets": {
    "1": 125,
    "2": 1842,
    "3": 3620,
    "4": 5418
  }
}
```

PoC에서는 처음부터 모든 시간을 자동으로 맞추지 않는다. 우선 각 쿼터 시작 영상 시점만 수동으로 찍고, 문자중계의 경기 시계와 영상 시간을 거칠게 연결한다. 이후 스코어보드 OCR을 붙여 정확도를 높인다.

#### 3.4.2 시간 매칭 문제

문자중계 시간과 영상 시간은 서로 다르다.

- 문자중계 시간: 경기 시계 기준. 예: 2Q 06:34
- 영상 시간: 실제 재생 시간 기준. 예: 영상 38분 12초

농구는 파울, 자유투, 작전타임, 비디오 판독, 선수 교체 때 경기 시계가 멈춘다. 따라서 단순히 `쿼터 시작 영상 시점 + 경기 시계 차이`로 계산하면 시간이 점점 어긋난다.

이를 해결하기 위한 매칭 기준은 다음과 같다.

1. 같은 쿼터
2. 비슷한 경기 시계
3. 점수 변화 일치
4. 이벤트 순서 일치
5. 스코어보드 OCR 결과와 문자중계 점수 일치

즉 영상 모델의 핵심 난이도는 "선수 인식"보다 "문자중계 이벤트와 영상 타임라인을 안정적으로 동기화하는 것"이다.

#### 3.4.3 하이라이트 컷팅 규칙

초기 규칙:

- 득점 성공 이벤트: 이벤트 시점 기준 `-10초 ~ +5초`
- 3점슛 성공: 이벤트 시점 기준 `-12초 ~ +6초`
- 덩크/블록/속공: 이벤트 시점 기준 `-8초 ~ +6초`
- 어시스트가 붙은 득점: 직전 패스/어시스트 이벤트까지 포함
- 자유투: 기본 하이라이트 후보에서 제외하되, 클러치 상황에서는 포함
- 파울: 득점으로 연결되는 앤드원 상황만 포함

예시:

```text
문자중계: 2Q 06:34 유기상 3점슛 성공
영상 매칭: 00:38:12 부근
클립 구간: 00:38:00 ~ 00:38:18
저장 결과: 유기상 3점 하이라이트 후보
```

#### 3.4.4 구현 컴포넌트

초기 PoC 구성:

```text
KBL 문자중계 파서
    ↓
득점/주요 이벤트 필터
    ↓
쿼터 시작점 수동 입력
    ↓
영상 시간 후보 계산
    ↓
스코어보드 OCR로 시간/점수 보정
    ↓
FFmpeg 클립 컷팅
    ↓
선수별 하이라이트 후보 JSON 저장
```

기술 후보:

- 문자중계 수집: KBL 웹 페이지/API 분석, `fetch`, `Playwright`, `cheerio`
- 이벤트 정규화: Node.js 또는 Python
- 영상 컷팅: FFmpeg
- 프레임 샘플링: OpenCV
- 스코어보드 OCR: PaddleOCR, EasyOCR, Tesseract, 또는 OCR API
- 후보 저장: JSON, PostgreSQL, S3 호환 스토리지

#### 3.4.5 PoC 성공 기준

1개 경기 기준으로 다음을 달성하면 1차 성공으로 본다.

- 문자중계에서 득점 이벤트 목록을 구조화한다.
- 쿼터 시작 영상 시점을 입력한다.
- 득점 이벤트 기준으로 하이라이트 후보 클립을 자동 생성한다.
- 사람이 확인했을 때 70% 이상이 해당 득점 장면을 포함한다.
- 선수별로 클립을 묶어 팬 앱에 전달할 metadata를 만든다.

OCR을 붙인 2차 PoC의 성공 기준:

- 경기 시계/점수 OCR로 이벤트 매칭 시간을 보정한다.
- 득점 장면 포함 정확도를 85% 이상으로 올린다.
- 리플레이/광고/타임아웃 구간을 자동으로 피하는 규칙을 만든다.

#### 3.4.6 모델 개발 난이도 재정의

이 접근에서는 1차 개발물을 "딥러닝 영상 모델"보다 "하이라이트 추출 파이프라인"으로 정의한다.

| 기능 | 1차 PoC 난이도 | 제품화 난이도 | 비고 |
| --- | --- | --- | --- |
| KBL 문자중계 파싱 | 중간 | 중간 | 페이지/API 구조 파악 필요 |
| 득점 이벤트 필터링 | 낮음 | 낮음 | 규칙 기반 가능 |
| 쿼터 시작점 수동 매핑 | 낮음 | 낮음 | PoC에서는 수동 입력 허용 |
| 경기 시계 OCR | 중간 | 높음 | 영상 화질/스코어보드 UI 영향 |
| 이벤트-영상 매칭 | 중간 | 높음 | 경기 시계 정지 때문에 보정 필요 |
| FFmpeg 자동 컷팅 | 낮음 | 중간 | 타임코드만 맞으면 구현 쉬움 |
| 선수 자동 인식 | 높음 | 매우 높음 | 초기에는 문자중계 선수명으로 대체 |

따라서 CourtLens의 첫 영상 AI 목표는 "영상만 보고 선수를 식별하는 모델"이 아니라, "공식 문자중계와 영상 타임라인을 결합해 선수별 하이라이트 후보를 자동 생성하는 모델/파이프라인"으로 잡는다.

### 3.5 모델 구성 후보

MVP 단계:

- play-by-play 기반 이벤트 구간 추출
- 경기 시간 OCR 또는 수동 타임코드 매핑
- OpenCV 기반 컷 분할
- 선수 인식은 완전 자동화하지 않고 로스터/play-by-play로 우선 매핑
- KBL 문자중계 이벤트를 기준으로 득점/어시스트/파울 이벤트를 구조화
- FFmpeg로 이벤트 전후 구간을 자동 컷팅
- 사람이 검수 가능한 하이라이트 후보 생성

제품화 단계:

- Object detection: YOLO 계열, RT-DETR 계열
- Tracking: ByteTrack, DeepSORT 계열
- OCR: PaddleOCR, EasyOCR, 또는 경기 스코어보드 특화 OCR
- Action recognition: VideoMAE, SlowFast, TimeSformer 계열
- Temporal action localization: 이벤트 시작/종료 구간 자동 보정
- Highlight ranking: 장면 통계, 팬 반응, 선수 관심도를 결합한 랭킹 모델

### 3.6 학습/라벨링 데이터

필요 라벨:

- 경기 ID
- 쿼터와 경기 시간
- 이벤트 타입
- 관련 선수
- 관련 팀
- 시작/종료 프레임
- 하이라이트 여부
- 장면 품질 점수
- 팬 앱 추천 가능 여부

라벨링 방식:

- 1차: play-by-play와 박스스코어 기반 자동 라벨링
- 2차: 사람이 주요 장면 후보를 검수
- 3차: 팬 앱 반응 데이터로 좋은 하이라이트 기준 재학습

초기에는 완전한 지도학습보다 rule-based + weak labeling 방식이 현실적이다. 영상 권리와 라벨링 비용이 크기 때문에, MVP에서는 소량 경기로 검증하고 제품화 단계에서 구단 협업 데이터를 확보한다.

### 3.7 평가 지표

모델 품질:

- 이벤트 탐지 정확도
- 선수 매칭 정확도
- 시작/종료 타임코드 오차
- 하이라이트 후보 precision/recall
- 클립 품질 점수

서비스 지표:

- 하이라이트 생성 완료율
- 클립 조회율
- 저장/공유율
- 작전판AI 열람률
- 직관메이트/예매/굿즈 CTA 클릭률

## 4. 모델 2: 구단 이벤트/마케팅 추천 LLM

### 4.1 목표

팬 행동 로그와 댓글 반응, 선수 맥락, 경기 맥락을 바탕으로 구단이 실행할 수 있는 이벤트와 마케팅 액션을 제안한다.

이 모델은 단순 문장 생성기가 아니라 구단 대시보드의 AI Strategy Engine 역할을 한다.

최종 산출물:

- Revenue Opportunity Top 3
- 이벤트 브리프
- 팬 앱 이벤트 카드 초안
- 스폰서 패키지 제안서 초안
- Performance 개선 액션
- 예상 KPI와 추적 지표

### 4.2 입력 데이터

팬 행동 데이터:

- 선수 선택

- 저장/공유
- 작전판AI 열람
- 입덕패스 조회

- 예매 CTA 클릭
- 굿즈 CTA 클릭
- 이벤트 CTA 클릭

댓글/텍스트 데이터:

- 팬 앱 댓글
- Fan Voice 의견
- 이벤트 투표 응답
- 좌석/굿즈/경기장 경험 관련 피드백
- SNS/커뮤니티 반응

선수/경기 맥락:

- 선수 포지션
- 최근 폼

- 기록 달성 여부
- 부상/복귀 예정
- 프랜차이즈 스타 여부
- 다음 홈/원정 경기

구단 운영 맥락:

- 이벤트 캘린더
- 굿즈 재고
- 좌석 판매 현황
- 스폰서 카테고리
- 금지 문구와 브랜드 톤
- 구단 마케팅 우선순위

### 4.3 처리 파이프라인

1. 데이터 집계
   - 팬 행동 로그를 선수/장면/경기/세그먼트 기준으로 집계
   - 댓글과 Fan Voice를 카테고리별로 분류
   - 개인정보를 제거하고 집계 데이터 중심으로 사용

2. 반응 분석
   - 어떤 선수가 반응이 높은지 계산
   - 어떤 장면이 저장/공유/댓글을 많이 유도했는지 계산
   - 직관메이트, 예매, 굿즈 CTA까지 이어지는 경로 분석

3. 기회 후보 생성
   - 단기성 이벤트 후보
   - 라운드 이벤트 후보
   - 장기 프랜차이즈 마케팅 후보
   - 스폰서 패키지 후보

4. 후보 점수화
   - 팬 반응 강도
   - 예상 실행 난이도
   - 예상 매출 기여
   - 구단 운영 가능성
   - 스폰서 적합도

5. LLM 생성
   - Top 3 추천 이유 생성
   - 이벤트명, 대상 팬 세그먼트, CTA 문구, 노출 위치 생성
   - 스폰서 제안 문구 생성
   - Performance 추적 KPI 생성

6. 검증과 저장
   - JSON schema 검증
   - 금지어/과장 표현 필터링
   - 근거 지표 누락 여부 확인
   - 구단 담당자 승인 전에는 팬 앱에 자동 게시하지 않음

### 4.5 모델 구성 후보

## 5. 두 모델의 연결 지점

영상 모델과 LLM 모델은 독립적으로 보이지만 실제 제품에서는 강하게 연결된다.

| 연결 데이터 | 영상 모델 역할 | LLM 모델 역할 |
| --- | --- | --- |
| 선수별 클립 | 선수와 장면을 태깅 | 팬 반응 높은 선수 이벤트 추천 |
| 장면 유형 | 3점, 리바운드, 어시스트 등 분류 | 장면에 맞는 이벤트/굿즈/스폰서 문구 생성 |
| 타임코드 | 클립 시작/종료 제공 | 스폰서 제안서에 대표 장면 근거로 첨부 |
| 팬 반응 | 클립별 조회/저장/공유 로그 생성 | Revenue Opportunity 점수화 |
| 작전판AI 태그 | 전술 설명용 맥락 제공 | 입덕패스/이벤트 카피 생성 |

핵심은 영상 모델이 만든 "선수별 하이라이트 자산"이 LLM 모델의 "구단 실행 전략"으로 이어지는 것이다.

## 6. 백엔드 연동 구조

제품화 시 백엔드는 두 모델을 직접 프론트에서 호출하지 않게 관리한다.

### 주요 API

```text
POST /api/highlights/jobs
GET  /api/highlights/jobs/{job_id}
GET  /api/highlights/clips?game_id=&player_id=

POST /api/ai/revenue-opportunities
POST /api/ai/event-briefs
POST /api/ai/sponsor-packages
POST /api/ai/performance-reports
```

### 비동기 처리

영상 분석은 시간이 오래 걸리므로 queue 기반으로 처리한다.

- 요청 생성
- 영상 분석 job 등록
- 진행률 저장
- 완료 후 clip metadata 저장
- 팬 앱에서 polling 또는 websocket으로 완료 알림

LLM 추천은 상대적으로 짧지만 구단 데이터와 로그 집계가 필요하므로 서버에서 실행한다.

- 집계 데이터 생성
- 프롬프트/RAG 컨텍스트 구성
- LLM 호출
- JSON 검증
- GeneratedAsset 저장

## 7. 보안/권리/운영 리스크

영상 권리:

- KBL 경기 전체 영상은 저작권과 중계권 이슈가 있다.
- 실제 제품에서는 구단 제공 영상, 공식 클립, 권리 계약이 된 VOD만 사용해야 한다.
- MVP에서는 샘플 영상 또는 공개 사용 가능한 짧은 클립으로만 검증한다.

개인정보:

- 팬 클릭/댓글 데이터는 개인 식별자가 아니라 익명화/집계 형태로 분석한다.
- 구단 대시보드에는 개인 단위가 아니라 세그먼트 단위로 보여준다.

모델 신뢰성:

- 영상 모델은 선수 오인식 가능성이 있다.
- LLM은 근거 없는 추천을 만들 수 있다.
- 모든 추천에는 근거 지표를 붙이고, 구단 담당자 승인 단계를 둔다.

운영 리스크:

- 자동 게시 금지
- 금지어/브랜드 세이프티 필터
- 추천 근거 로그 저장
- 모델 출력 버전 관리

## 10. 발표에서 설명할 핵심 문장

CourtLens의 AI는 단순히 영상을 자르는 모델이 아니다.

첫 번째 모델은 KBL 문자중계/play-by-play와 전체 경기 영상을 연결해 선수별 하이라이트 자산을 만든다. 처음부터 모든 선수를 영상만으로 인식하는 것이 아니라, 공식 문자중계의 선수/이벤트 정보를 기준으로 영상 타임라인을 맞추고 득점 장면을 자동 컷팅하는 방식으로 시작한다.

두 번째 모델은 그 하이라이트에 대한 팬 반응과 댓글을 분석해 구단이 바로 실행할 수 있는 이벤트, 마케팅, 스폰서 제안으로 바꾼다.

즉 CourtLens는 "AI 하이라이트 생성"에서 끝나는 서비스가 아니라, 팬이 재미있게 본 장면을 구단의 매출 액션으로 연결하는 AI 운영 플랫폼이다.
